# PART 1. 프로젝트 개요

## 1.1 프로젝트 목적

말레이시아 RFID/ANPR 기반 SLFF(Single Lane Free Flow) 및 MLFF(Multi-Lane Free Flow) Tolling 시스템의 Back Office System(BOS)을 구축하여 말레이시아 고속도로 시장에서 수주를 확보하고, 장기적으로 AI·빅데이터·블록체인 기반의 차세대 Tolling 플랫폼으로 발전시키는 것을 목표로 한다.

# PART 2. 말레이시아 시장 현황 및 사업 기회

## 2.1 말레이시아 Tolling 시장 현황 (2025~2026)

### 주요 플레이어 현황

| 구분 | 플레이어 | 기술 | 현황 |
|---|---|---|---|
| TnG | Titan Flow MLFF | RFID Primary + ANPR + LiDAR + AI | 2025.10 POC 완료 |
| PLUS | JustGo | ANPR Primary + 계좌 직불 | 2025.10 13개 플라자 Pilot |
| 기타 | 4개 컨소시엄 | 혼합 방식 | 입찰 경쟁 중 |

### Tolling 전환 로드맵

```
현재 (RFID 배리어)  →  SLFF  →  MLFF
  30km/h 제한          50km/h     고속 다차로 자유 통과
  배리어 있음          배리어 없음   완전 개방
  RFID 전용            RFID + ANPR   RFID + ANPR + LiDAR
```

### 수주 기회 분석

말레이시아 정부는 33개 고속도로 콘세셔네어가 B2B 모델로 파트너를 직접 선택할 수 있도록 했으며 TnG·PLUS 외 6개 컨소시엄이 MLFF 계약을 놓고 경쟁하고 있다. SLFF는 현재 운영 중인 RFID 인프라의 즉시 업그레이드 수요가 발생하고 있으며, SLFF BOS를 선납품하고 동일 BOS에서 MLFF 전환 스위치를 제공함으로써 차별화가 가능하다.

# PART 3. BOS 서비스 도메인 상세 정의

## 3.1 SLFF vs MLFF BOS 주요 차이

| 기능 | SLFF BOS | MLFF BOS |
|---|---|---|
| 차로 처리 | 단일 차로, 배리어 없음 | 다차로 동시 처리 |
| RFID 처리 | 단일 태그 읽기 | 다차로 동시 다중 태그 |
| ANPR 역할 | RFID 미탑재 차량 보조 | 모든 차량 2차 검증 |
| 진입/진출 매칭 | 불필요 (단일 요금소) | 필수 (구간 요금 계산) |
| 처리 성능 | 1,000~3,000 TXN/시/차로 | 10,000+ TXN/시/전체 |
| 미납 처리 | 단순 미결제 추적 | 진입 후 미진출 포함 |

## 3.2 BOS 전체 서비스 도메인 (8개)

### Domain 1: Transaction Processing Engine

**RFID 트랜잭션 처리 흐름:**
```
RFID Tag 수신 → Tag ID 유효성 검증
→ 차량 정보 조회 (Redis Cache → PostgreSQL)
→ 차종 판별 → 요금 계산 (차종·구간·시간대·할인)
→ TnG eWallet 차감 요청 → 결과 수신
→ Transaction DB 저장 → 영수증 발행
→ Kafka Producer → 빅데이터 레이어 전송
→ 실시간 집계 업데이트
```

**ANPR 트랜잭션 처리 흐름:**
```
ANPR 이미지 수신 → 외부 OCR 엔진 호출
→ 번호판 텍스트 추출 → JPJ DB 조회
→ 계정 매칭 (JustGo/TnG 연동)
→ 매칭 성공: 결제 처리
→ 매칭 실패: Violation 생성 (V001)
→ 이미지 S3 암호화 저장
```

**MLFF 전용 — 진입/진출 매칭:**
```
Entry Event 수신
→ Redis에 Session 생성
   Key: {RFID TagID or 번호판}
   Value: {진입 플라자, 진입 시각, 차종}
   TTL: 24시간

Exit Event 수신
→ Redis에서 Entry Session 조회
→ 구간 요금 계산 (진입~진출 거리 기반)
→ Session 종료 → Transaction 확정

Timeout (미진출):
→ 최장 구간 요금 부과 or Violation(V005) 생성
```

**성능 목표:**
- RFID 처리: < 100ms per transaction
- ANPR 연동: < 500ms per image
- 동시 처리: 10,000 TPS (MLFF 피크)
- 시스템 가용성: 99.99% (4-Nine)

```
  콘세셔네어별 비교 (소속 도로만)
```

### Domain 4: Violation & Enforcement

**위반 유형 분류:**
```
V001: RFID 미탑재 통과 (ANPR으로만 감지)
V002: 잔액 부족 통과 (결제 실패)
V003: 번호판 인식 실패 (오염/훼손 번호판)
V004: 번호판 클론 의심 (JPJ DB 불일치)
V005: MLFF 진입 후 미진출 (도주 의심)
V006: 차량 등급 불일치 (오토바이→승용차 차선)
```

**위반 처리 워크플로:**
```
위반 감지
→ 증거 수집 (ANPR 이미지 + RFID 로그 + 타임스탬프)
→ AI 자동 분류 (V001~V006)
→ JPJ 소유자 조회
→ 고지서 자동 생성 (Bahasa Melayu + English)
→ RPA 자동 발송 (이메일·SMS·우편)
→ 납부 대기 (30일)
→ 납부 완료: 케이스 종결
→ 미납 (30일 초과): Unpaid Management 이관
→ 이의신청 접수: 워크플로 분기
→ 처리 → 최종 결정 → 통보

[법적 근거]
Federal Roads (Private Management) Act 1984 Section 8
벌금: RM 2,000 ~ RM 5,000
향후 형사처벌 조항 추가 시 자동 반영 (모듈 설계)
```

## 5.4 RBAC 구현 기술 설계

```sql
-- PostgreSQL Row-Level Security 예시
-- 조직 기반 자동 데이터 필터링

-- TOC 데이터 격리
CREATE POLICY toc_data_isolation ON transactions
  USING (
    highway_id IN (
      SELECT highway_id FROM user_highway_access
      WHERE user_id = current_user_id()
    )
  );

-- JVC 전체 접근
CREATE POLICY jvc_full_access ON transactions
  USING (
    (SELECT org_type FROM users
     WHERE id = current_user_id()) = 'ORG_JVC'
  );

-- Plaza 자체 데이터만
CREATE POLICY plaza_self_only ON transactions
  USING (
    plaza_id = (
      SELECT plaza_id FROM users
      WHERE id = current_user_id()
    )
  );
```

# PART 6. 혁신 서비스 기획 — 결론: 전문 기획 인력 필수

## 6.1 왜 전문 기획 Agent가 필요한가

기술팀(CTO)이 할 수 없는 영역이 존재한다. 요금 변경 시 통행량·수입 변화 예측, 교통 수요 탄력성(Price Elasticity) 분석, 시간대별 동적 요금제 수익 최적화 설계, 신규 서비스 비즈니스 모델 설계가 그것이다. 또한 CPO(제품책임자)가 할 수 없는 영역도 있다. 교통공학적 시뮬레이션 설계, 빅데이터 분석 방향의 도메인 기반 제시, Tolling 산업 규제·정책 변화 대응 기획, 콘세셔네어별 수익 최적화 컨설팅이 이에 해당한다.

따라서 교통/Tolling 도메인 전문 기획 전담 Division이 독립적으로 필요하다.

## 6.2 혁신 서비스 발굴 포트폴리오

### 시뮬레이션 서비스

```
[S01] 요금 변경 시나리오 시뮬레이션
  입력: 요금 변경률(%), 차종, 구간, 시간대
  처리: 교통 수요 탄력성 계수 적용
        말레이시아 탄력성: -0.3 ~ -0.5
        (요금 10% 인상 시 통행량 3~5% 감소)
  출력: 통행량 변화율, 수입 변화액, 손익분기점
  예시: "요금 10% 인상 → 통행량 -4% →
         수입 순증가 +6%"

[S02] 동적 요금제 수익 최적화 시뮬레이션
  입력: 시간대별 요금 구간 설계안
  처리: 과거 통행 패턴 기반 시뮬레이션
  출력: 최적 요금 조합, 예상 수익 증가율
  예시: "출퇴근 +20%, 심야 -30% 조합
         → 연간 수입 +8.5% 예상"

[S03] 신규 노선 개통 영향 분석
  "A도로 개통 시 기존 B도로 통행량 변화 예측"

[S04] 이벤트 기반 시뮬레이션
  "하리 라야 50% 할인 시 통행량·수입·서비스 부하 시뮬레이션"

[시뮬레이션 출력 포맷]
{
  scenario: "요금 10% 인상 — Class 2 전체",
  current_daily_volume: 150,000 대,
  predicted_volume: 144,000 대,
  volume_change: -4.0%,
  current_daily_revenue: RM 180,000,
  predicted_revenue: RM 190,080,
  revenue_change: +RM 10,080 (+5.6%),
  breakeven_period: "7일",
  confidence_interval: "±2.1%"
```

# PART 7. Paperclip 조직 구성 v3.0

## 7.1 Paperclip 핵심 개념

Paperclip은 AI Agent들을 실제 회사처럼 조직화하여 운영하는 오픈소스 플랫폼이다. Board(사람)는 이사회 역할로 전략 승인과 예산 결정 및 주요 의사결정을 담당하고, Agent들은 조직 계층 구조 안에서 자율적으로 작업을 수행한다.

핵심 운영 메커니즘은 Heartbeat로, Agent는 상시 실행이 아니라 스케줄·태스크 할당·코멘트·수동 실행 트리거로 깨어나서 작업 후 상태를 업데이트한다. Skills는 Agent가 필요할 때만 로드하는 Markdown 파일로 Tolling 도메인 전문 지식을 Agent에게 주입한다. Governance는 고위험 작업(Agent 채용, CEO 전략, 예산 변경)에 Board 승인을 요구한다.

```
[일일 5~10분]
```
