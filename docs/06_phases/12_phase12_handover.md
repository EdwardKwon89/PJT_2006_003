## 1. Phase 개요

### 1.1 목적 및 범위

Phase 12는 Malaysia SLFF/MLFF Tolling BOS 프로젝트의 **최종 인수인계 & 운영 이관** 단계다. 전체 12개 Phase에 걸쳐 구축된 시스템의 문서를 최종 확정하고, JVC 운영팀에 교육과 실습을 제공하며, 프로덕션 시스템을 공식적으로 이관하고 하자보수 기간(Warranty)을 시작한다.

**핵심 목표:**
- 전체 문서(46개 설계 문서 + 21개 Skills) 최종 검토 및 확정
- 운영팀 교육 3단계 완료 (기본 → 실습 → 자격 인증)
- 프로덕션 이관 체크리스트 완료
- RunBook 기반 온콜 이관 완료
- 하자보수 기간 (90일) 지원 체계 수립

### 1.2 Phase 연계

| 구분 | 내용 |
|------|------|
| **선행 Phase** | Phase 1~11 전체 완료 필수 |
| **예상 기간** | **4주** (Sprint 29~32) |
| **특이사항** | Phase 12 완료 = 프로젝트 공식 종료 |

### 1.3 인수인계 타임라인

```
Week 1: 문서 최종 확정 & 검토
  D1~2: 46개 설계 문서 최종 검토
  D3~4: 21개 Skills 파일 검토
  D5:   문서 버전 동결 (v1.0 태그)

Week 2: 운영팀 교육 (기본 & 실습)
  D8~9:  기본 교육 (시스템 개요, 모니터링, 일반 운영)
  D10~11: 실습 교육 (장애 대응, RunBook 실행)
  D12:   교육 평가 & 자격 인증 시험

Week 3: 시스템 핸드오버 & 이관
  D15~16: Dry Run 이관 테스트
  D17:   공식 이관 (Go-Live Sign-off)
  D18~19: 이관 후 집중 모니터링 기간

Week 4: 안정화 & 하자보수 체계 수립
  D22~25: 잔여 이슈 처리
  D26:   최종 프로젝트 완료 보고서
  D28:   프로젝트 공식 종료 미팅
```

## 2. 담당 Agent 및 역할

| Agent | 역할 | 주요 책임 |
|-------|------|---------|
| `PM` | 프로젝트 매니저 | 인수인계 계획 총괄, 이해관계자 조율 |
| `CTO` | 기술 리드 | 기술 문서 최종 검토, 이관 승인 |
| `devops-lead` | 인프라 담당 | 프로덕션 이관 실행, 온콜 이관 |
| `Compliance` | 컴플라이언스 | 최종 컴플라이언스 확인서 작성 |
| 전체 도메인 Lead | 도메인별 문서 담당 | 담당 문서 최종 검토 및 서명 |

## 3. 주요 태스크 체크리스트

### 3.1 문서 최종 확정

- [ ] **설계 문서 46개 최종 검토 체크리스트**
  - 각 Lead Agent 자신의 담당 문서 내용 정확성 확인
  - 상호 참조 링크 유효성 최종 확인 (Python 링크 검증 스크립트 재실행)
  - 버전 정보 v1.0 확정 및 날짜 업데이트
- [ ] **21개 Skills 파일 최종 검토**
  - 각 Skills의 `SKILL.md` 내용 동작 검증
  - Skills 간 의존성 맵 작성
- [ ] **문서 버전 동결**
  - Git 태그 `v1.0.0` 생성 (`git tag -a v1.0.0 -m "Phase 12: Final Documentation Freeze"`)
  - GitHub Release 노트 작성

### 3.2 운영팀 교육

- [ ] **기본 교육 (8시간)**
  - BOS 시스템 전체 개요 및 아키텍처 설명
  - 일별 운영 절차 (배치 확인, 모니터링 확인, 이상 보고)
  - Grafana 대시보드 사용법
  - 자주 발생하는 일상 태스크 실습
- [ ] **장애 대응 실습 (8시간)**
  - 10개 RunBook 시나리오 실습 (Phase 7 RunBook 기반)
  - 실제 Staging 환경 장애 시뮬레이션
  - PagerDuty 온콜 대응 절차 실습
- [ ] **자격 인증 시험**
  - 필기: 30문항 (시스템 구조, RunBook, 에스컬레이션 절차)
  - 실기: Staging 환경 장애 대응 (2시나리오)
  - 합격 기준: 필기 80% + 실기 Pass
  - 불합격 시 재교육 1회 후 재시험 허용

### 3.3 프로덕션 이관 (Go-Live)

- [ ] **이관 전 체크리스트**
  - [ ] 모든 Phase (1~11) 완료 확인
  - [ ] 프로덕션 환경 최종 배포 완료
  - [ ] SLO 기준 24시간 안정화 확인 (에러율 < SLO 임계값)
  - [ ] 백업 및 롤백 계획 수립 완료
  - [ ] 운영팀 교육 자격 인증 완료
  - [ ] 컴플라이언스 담당자 서명 완료
  - [ ] CTO 최종 승인 서명 완료
- [ ] **이관 실행 (D Day)**
  - 이관 시작 선언 (운영 이관 회의)
  - `devops-lead` → JVC 운영팀 온콜 이관
  - DNS, 방화벽, VPN 접근 권한 이관
  - 모니터링 알림 수신자 JVC 운영팀으로 변경
  - 이관 완료 서명 (`프로젝트 완료 확인서`)
- [ ] **이관 후 집중 모니터링 (72시간)**
  - 개발팀 대기 (이관 후 72시간 지원)
  - 긴급 이슈 발생 시 2시간 내 대응

### 3.4 하자보수 체계 (Warranty 90일)

- [ ] **하자보수 계약 조건 정의**
  - 기간: 이관일로부터 90일
  - 범위: Phase 1~12 구현 범위 내 결함 수정
  - 제외: 신규 기능 개발, 운영 환경 변경으로 인한 이슈
  - SLA: P1 (Critical) 4시간 내 / P2 (High) 24시간 내 / P3 8 영업일 내
- [ ] **하자보수 지원 채널 설정**
  - 전용 Slack 채널: `#bos-warranty-support`
  - 이슈 트래커: GitHub Issues (별도 Warranty 레이블)
  - 온콜: 개발팀 원래 온콜 스케줄 유지 (90일)

### 3.5 최종 프로젝트 완료 보고서

- [ ] **프로젝트 성과 요약 보고서**
  - 12개 Phase 완료 현황 및 주요 성과
  - 구축된 시스템 규모 (서비스 수, 문서 수, 코드 라인)
  - KPI 달성 현황 (처리량, 가용성, 비용)
  - 프로젝트 기간 및 예산 집행 현황
- [ ] **Lessons Learned 문서**
  - 잘된 점, 개선이 필요한 점, 다음 프로젝트에 적용할 사항
- [ ] **미래 로드맵 제안**
  - Phase 13+ 잠재적 개선 사항
  - 기술 부채 목록 (해결 우선순위 포함)
