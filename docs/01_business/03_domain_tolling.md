## 개요

통행료 징수(Tolling) 도메인은 Malaysia BOS(Back-Office System)의 핵심 비즈니스 영역으로, **차량이 요금도로(toll highway)를 통행할 때 발생하는 요금을 징수하고 관리하는 모든 프로세스**를 포괄한다. 이는 기술(RFID/ANPR 센서)과 비즈니스 로직(요금 계산, 결제 매칭, 미결제 처리)의 교집합이다.

### 핵심 특징

- **원활한 통행(Seamless Flow)**: 차량이 요금소를 지나칠 때 정차하지 않음
- **다중 인증 기술**: RFID 태그 + ANPR(자동번호판인식) + 결제 수단 조합
- **실시간 결제 처리**: Transaction Engine이 10,000 TPS 수준에서 처리
- **거버넌스 복합성**: JVC(사업 주체), TOC(운영사), Clearing Center(정산) 등 다층 조직 관여

## 전통 통행료 징수 체계

### 1. 폐쇄식 (Closed) 시스템

**정의**: 이용자가 요금도로 진입 지점(Entry)에서 통행권을 받고, 퇴출 지점(Exit)에서 반납 및 요금 결제

#### 특징

| 항목 | 설명 |
|------|------|
| **통행권(Ticket)** | 종이 또는 자기카드 형태, Entry에서 발급 |
| **유인 요금소** | 상담원이 상주하며 통행권 발급/회수 |
| **요금 계산** | Exit 지점에서 Entry-Exit 거리에 따라 즉석 계산 |
| **지연 요소** | 통행권 수령, 결제 프로세스로 인한 정체 |
| **신뢰도** | 사람 개입으로 인한 오류 가능성 |

#### 프로세스 흐름

```
[Entry] → 통행권 수령 → 주행 → [Exit] → 요금 확인 및 결제 → 통행권 반납 → 통과
```

**Malaysia 사례**: PLUS(주요 고속도로) 일부 구간에서 여전히 폐쇄식 또는 혼용 모드 운영

---

### 2. 개방식 (Open) 시스템

**정의**: Entry/Exit 정보 없이 통행한 거리나 구간에 따라 일정 기간 후 청구

#### 특징

| 항목 | 설명 |
|------|------|
| **통행권** | 없음 (거리 기반 청구) |
| **비인원화** | 자동 인식 시스템만 운영 |
| **요금 계산** | 사후 청구 방식, 통행 거리 누적 후 청구 |
| **지연 요소** | 없음 (차량 정차 제로) |
| **신뢰도** | 기술 오류 시 청구 오류 발생 가능 |

#### 프로세스 흐름

```
[주행] → RFID/ANPR 자동 인식 → 로그 기록 → [사후 청구] → 결제
```

---

### 3. 유인 요금소 (Toll Plaza)

말레이시아 전통 통행료 징수의 중심 구조.

#### 역할

| 역할 | 책임 |
|------|------|
| **요금 징수** | 현금, 카드, TnG 등 다양한 결제 수단 지원 |
| **통행권 관리** | 폐쇄식 시스템에서 Ticket 발급/회수 |
| **실시간 거래** | 차량 정차 중에 결제 완료 및 통행 허가 |
| **오류 대응** | 태그 인식 실패, 결제 거절 등 즉시 처리 |
| **현황 보고** | 일일 거래액, 차량 통행량 집계 |

#### 기술 구성

```
[RFID Reader] → 태그 판독 → [결제 시스템]
     ↓                           ↓
 [ANPR Camera]           [POS Terminal]
     ↓                           ↓
  번호판 인식 → [Toll Plaza Computer] → [결제 게이트웨이]
```

## 말레이시아 JVC 모델                                                                                               

Malaysia 통행료 징수는 **JVC(Joint Venture Company) 기반 거버넌스**로 운영되며, 각 도로별로 별도의 JVC가 사업권을 보유한다.

### 1. 조직 계층 구조

```
┌─────────────────────────────────────────┐
│         JVC (사업 주체)                  │
│   (예: PLUS, PKLRH, MKIJA 등)          │
└──────────────┬──────────────────────────┘
               │
       ┌───────┼───────┐
       │       │       │
   ┌───▼───┐ ┌─▼────┐ ┌─▼──────┐
   │ TOC   │ │BOS   │ │Finance │
   │(운영) │ │(관리)│ │(정산)  │
   └───┬───┘ └──────┘ └────────┘
       │
   ┌───▼────────────────┐
   │ 산하 구조           │
   │ - Toll Center      │
   │ - Toll Plaza       │
   │ - Service Area     │
   │ - Clearing Center  │
   └────────────────────┘
```

### 2. 각 조직의 역할

#### 2.1 JVC (Joint Venture Company)

**정의**: 정부와 민간이 공동으로 출자한 도로 운영 회사

**책임**:
- 도로 운영권 보유 및 사업 전략 수립
- TOC(운영), BOS(관리), Finance(정산) 감시
- 이용자 고충 처리 및 공공 관계

**보유 자산**:
- 요금 도로 전체 운영 권한
- 모든 Toll Center, Plaza, Clearing Center 지배 권한
- 수입금 관리

---

#### 2.2 TOC (Traffic Operation Center)

**정의**: 도로 운영 및 차량 통행의 실시간 관리

**책임**:
- Toll Plaza, Toll Center 운영 감시
- 실시간 차량 흐름 모니터링
- 사고/장애 대응 및 우회로 안내
- 장비 고장 신속 대응

**보유 권한**:
- 요금소 운영 일정 결정
- 차선 개방/폐쇄 권한
- 긴급 상황 대응 명령권

**데이터 접근**:
- 실시간 차량 통행 데이터 (읽기만)
- Lane 센서 상태 데이터
- 현황 대시보드

---

#### 2.3 BOS (Back-Office System)

**정의**: 통행료 징수의 모든 거래, 계정, 청구 관리

**책임**:
- Transaction 기록 및 관리
- Account(고객 계정) 관리
- 청구 및 결제 처리
- 위반/미결제 관리
- 장비 자산 관리
- 보고서 생성

**보유 권한**:
- 모든 거래 데이터 접근
- 계정 정보 조회/수정
- 청구 및 미결제 결정
- 감면 승인

**주요 모듈**:
- Transaction Engine: 실시간 거래 처리 (10,000 TPS)
- Billing Engine: 청구 및 정산
- Compliance Engine: 위반 등록 및 JPJ 통지

---

#### 2.4 Clearing Center (정산소)

**정의**: JVC와 외부 결제 기관(TnG, 은행, FPX) 간 자금 정산

**책상**:
- TnG 잔액 조회 및 결제 확인
- 은행 이체 내역 매칭
- FPX 거래 확인
- JPJ(교통안전청) 위반금 수금 및 이관

**거래 흐름**:

```
[차량 통행] → [BOS 거래 기록] → [채널별 결제 처리]
                                    │
                  ┌─────────────────┼─────────────────┐
                  │                 │                 │
            [Channel A]         [Channel B]      [미결제]
            (RFID/ANPR)         (TnG 연동)        (후속 청구)
                  │                 │                 │
                  └─────────────────┼─────────────────┘
                                    │
                    [Clearing Center] → 정산 및 이관
```

---

### 3. Toll Center vs Toll Plaza 구분

| 항목 | Toll Center (중앙) | Toll Plaza (현장) |
|------|-------------------|-------------------|
| **위치** | 고속도로 출구 부근 | 요금소 (Entry/Exit) |
| **기능** | 거래 처리, 보고서 생성 | 차량 정차, 현금 거래 |
| **인원** | 행정/IT 인력 | 상담원 (유인) |
| **BOS 접근** | 광범위 (모든 모듈) | 제한적 (POS, 간단 조회) |
| **책임** | 거래 확정, 청구 결정 | 현장 거래 즉시 처리 |

## 단일 차선 무정지 통행 (SLFF)

### 1. 개념

**SLFF (Single Lane Free Flow)**: 단일 차선에서 차량이 정차하지 않고 통행하며, RFID/ANPR 기술로 자동 인식 및 결제 처리

### 2. 기술 구성

```
         차량 진입
            │
            ▼
   ┌─────────────────────┐
   │  RFID Reader (10m)  │
   │  - 800MHz 주파수    │
   │  - 5m 판독 거리     │
   │  - 태그 인식        │
   └────────┬────────────┘
            │
            ├─ 태그 있음 (90% 경우)
            │    ↓
            │ [Tag ID 검출]
            │    ↓
            │ [Account 조회]
            │    ↓
            │ [잔액 확인]
            │    ├─ 충분 → 즉시 차감
            │    └─ 부족 → 로그 기록
            │
            └─ 태그 없음 (10% 경우)
                 ↓
            [ANPR Camera]
                 ↓
            [번호판 인식]
                 ↓
            [국가 DB 조회]
                 ↓
            [후속 청구]
```

### 3. 핵심 모듈

#### 3.1 RFID/ANPR Interface

**RFID 태그 인식 흐름**:

| 단계 | 작업 | 실패 시나리오 |
|------|------|--------------|
| 1 | Reader 범위 내 태그 신호 감지 | 태그 손상, 배터리 고갈 |
| 2 | 800MHz 주파수로 Tag ID 판독 | 간섭(다중 태그), 물리적 손상 |
| 3 | Transaction DB에 Tag ID 조회 | 미등록 태그, DB 오류 |
| 4 | Account 잔액 확인 | 계정 정지/해지 |
| 5 | 요금 차감 및 거래 기록 | 결제 게이트웨이 오류 |

**ANPR 번호판 인식 흐름**:

| 단계 | 작업 | 실패 시나리오 |
|------|------|--------------|
| 1 | 고속 카메라로 번호판 촬영 | 흐릿한 이미지, 각도 불량 |
| 2 | OCR(광학문자인식) 처리 | 더러운 번호판, 특수문자 |
| 3 | 국가 차량 DB 조회 | 미등록 차량, 외국 번호판 |
| 4 | 거래 로그 기록 | 번호판 매칭 실패 |
| 5 | 사후 청구 (Channel B 또는 미결제) | 주소 불명 |

---

#### 3.2 특이점 처리 (Edge Cases)

**시나리오 1: 태그 오류**
```
상황: RFID Reader가 Tag ID 판독 실패
원인: 손상된 태그, 배터리 고갈, 간섭
처리:
  - ANPR 자동 전환
  - 번호판 인식 후 사후 청구
  - 이용자 고충 처리 대기
```

**시나리오 2: OCR 실패**
```
상황: ANPR가 번호판 인식 실패
원인: 흐릿한 이미지, 가려진 번호판, 특수 차량
처리:
  - Manual Review 대기
  - 미결제 거래로 기록
  - 운영팀에 알림
```

**시나리오 3: 계정 잔액 부족**
```
상황: RFID 인식 완료, 하지만 계정에 잔액 없음
원인: 선불 방식에서 충전 미흡
처리:
  - 거래 거절 (Physical Gate 닫음)
  - 이용자에게 SMS 알림
  - Clearing Center에 미결제 기록
```

**시나리오 4: 결제 게이트웨이 장애**
```
상황: Network 지연으로 결제 처리 지연
원인: Payment Gateway 다운, Network Latency
처리:
  - 로컬 큐에 거래 임시 저장
  - 게이트 오픈 (차량 통과)
  - 나중에 일괄 처리 또는 수동 조정
```

---

### 4. SLFF 운영 모델

```
Channel A (RFID/ANPR 기반)
├─ RFID 태그 소유자
│  ├─ 선불 계정 (e.g., TnG 카드)
│  └─ 요금 즉시 차감
└─ ANPR 자동 인식 (태그 없는 경우)
   └─ 사후 청구 또는 미결제 등록

실시간 처리 흐름
────────────────
[차량 진입] → 0.2초 이내 인식 → 결제 프로세스 → [통과/거절]
```

## 다중 차선 무정지 통행 (MLFF)

### 1. 개념

**MLFF (Multi-Lane Free Flow)**: 여러 차선에서 동시에 차량이 정차 없이 통행하며, Entry와 Exit 매칭 로직으로 주행 거리 기반 요금 계산

### 2. 기술 차이점

| 항목 | SLFF | MLFF |
|------|------|------|
| **차선 수** | 1개 | 2개 이상 |
| **요금 기준** | 구간 단위 | Entry-Exit 거리 |
| **Session 개념** | 없음 (한 번의 거래) | 있음 (진입-퇴출 페어링) |
| **복잡도** | 낮음 | 높음 (매칭 로직) |
| **초기 투자** | 낮음 | 높음 |

### 3. Entry/Exit 매칭 로직

```
[Entry Point]
├─ 고속도로 구간 진입
├─ RFID/ANPR으로 차량 ID 기록
├─ Entry Time, Location 저장
└─ Session 시작

           ↓ 주행 중
           
[Exit Point]
├─ 고속도로 구간 퇴출
├─ 같은 차량 ID로 RFID/ANPR 재인식
├─ Exit Time, Location 저장
└─ Entry-Exit 페어링

           ↓ 매칭 엔진
           
[Matching Engine]
├─ Entry-Exit 시간차 검증 (이상 거리 감지)
├─ 통행 거리 계산 (Entry Location - Exit Location)
├─ 요금 계산 (km당 가격표 적용)
└─ 거래 확정
```

### 4. 차선 설비 모니터링

MLFF 운영을 위한 실시간 장비 상태 관리:

#### 4.1 Reader 장비

| 장비 | 기능 | 모니터링 항목 |
|------|------|--------------|
| **RFID Reader** | 태그 판독 | 신호 강도, 판독률, 배터리 |
| **ANPR Camera** | 번호판 촬영 | 화질, 조도, 프레임 레이트 |
| **센서 | 차량 검출 | 트리거율, 잘못된 감지 |
| **Gateway** | 물리적 게이트 | 개폐 상태, 구동 시간 |

#### 4.2 리포팅

```
Real-time Dashboard (TOC가 모니터링)
├─ Lane별 차량 통행량
├─ 평균 통행 속도
├─ 장비 상태 표시
│  ├─ GREEN: 정상
│  ├─ YELLOW: 주의 (성능 저하)
│  └─ RED: 장애 (즉시 대응)
├─ 미결제 거래 수
└─ Alert (장비 고장 등)
```

#### 4.3 예측 유지보수

```
AI-based Prediction
├─ RFID Reader 판독률 추세 분석
├─ ANPR 인식 정확도 저하 감지
├─ Gateway 구동 시간 증가 경보
└─ 배터리 고갈 예측 (센서류)

→ 사전 부품 교체 또는 정비 스케줄링
```

## 결제 채널 구조

Malaysia의 통행료 결제는 **Channel A** (RFID/ANPR 기반 후불)와 **Channel B** (TnG 선불)로 이원화되어 있다.

### 1. Channel A: Clearing Center 기반

**특징**: 직접 징수, 국가 차량 DB 연동

#### 1.1 흐름도

```
[RFID 또는 ANPR]
      ↓
[BOS Transaction Engine]
 - 거래 기록
 - 잔액 확인 (RFID의 경우)
 - 요금 계산
      ↓
[Billing Engine]
 - 청구서 생성 (ANPR의 경우)
 - 납기일 설정 (14일)
      ↓
[Clearing Center]
 - 차량 소유자 추적
 - 국가 DB (JPJ) 조회
 - 등기 편지 발송
      ↓
[결제 처리]
 - 은행 이체
 - 방문 납부
 - 온라인 결제
      ↓
[미결제 처리]
 - 납기 경과 시
 - JPJ 위반금 전환
 - 차량 검사 불합격 연계
```

#### 1.2 거래 유형

| 유형 | 설명 | 처리 시간 |
|------|------|---------|
| **RFID 선불** | 태그에 선충금된 잔액 차감 | 실시간 (<1초) |
| **RFID 후불** | 월 단위 청구 (선택 옵션) | 월별 정산 |
| **ANPR 일반** | 번호판 인식 후 청구 | 1~3일 (처리) |
| **ANPR 특수** | 중장비, 특수 차량 (경사로 등록) | 수동 처리 |

---

### 2. Channel B: TnG 연동

**특징**: 제3자 결제 중개, 실시간 이체

#### 2.1 TnG 통합 흐름

```
[RFID Reader: TnG 카드 인식]
      ↓
[RFID Transaction: Tag UID 추출]
      ↓
[TnG Bridge API]
 - 카드 검증
 - 잔액 확인
      ↓
[TnG 결제 프로세스]
 - TnG 시스템에 요청
 - 실시간 잔액 차감
      ↓
[응답]
 - 성공: 거래 확정
 - 실패: ANPR 자동 전환 또는 거절
      ↓
[정산]
 - JVC가 TnG에 정산 청구
 - TnG가 은행 계좌로 송금 (T+1)
```

#### 2.2 TnG 연동 특징

| 항목 | 설명 |
|------|------|
| **실시간성** | 거의 0.5초 (TnG 서버 응답 시간 포함) |
| **안정성** | TnG 시스템 장애 시 ANPR 자동 백업 |
| **수수료** | JVC에서 거래액의 1~2% 지불 |
| **정산 주기** | 일일 또는 주간 (TnG와 계약에 따라) |

---

### 3. 미결제 처리 프로세스

미결제 거래가 발생하는 경우의 처리 흐름:

```
[거래 발생] (ANPR, 잔액 부족 등)
      ↓
[BOS: 미결제 기록]
 - 거래 ID, 차량, 금액, 날짜
      ↓
[Billing Engine: 청구서 생성]
 - 1차 청구 (納期: 14일)
 - 이메일/SMS 알림
      ↓
[납기 경과 시]
     ┌─ 결제 → 거래 확정
     ├─ 미결제 → 상기장 발송
     └─ 무응답 → 2차 청구
            ↓
[Clearing Center: 법적 조치]
 - JPJ 통지 (위반금 전환)
 - 차량 검사 불합격 연계
 - 등기 편지 발송
      ↓
[최종 결과]
 - 결제: Account 확정
 - 미결제: 위반금 수금 (JPJ 통해)
 - 분할납부: 협의 (특별 경우)
```

#### 미결제 상태 추적

```
BOS Dashboard (미결제 관리)
├─ 미결제 거래 목록 (일자별, 차량별)
├─ 발송 상태 (초기, 1차 상기, 2차 상기, 법적조치)
├─ 결제 추적 (결제액, 적립일)
├─ 미결제 통계 (금액, 건수)
└─ JPJ 송부 현황 (위반금 변환된 건수)
```

## 기존 시스템 대비

### 1. HiPass (한국)와의 비교

| 항목 | Malaysia SLFF/MLFF | HiPass (한국) |
|------|-------------------|--------------|
| **운영 모델** | JVC 사업권 기반 | 도로공사 중앙 운영 |
| **요금 계산** | Channel A/B 이원화 | 통합 후불 |
| **결제 수단** | TnG, 은행, 현금 | 하이패스카드, 신용카드 |
| **인프라** | Toll Center/Plaza 분산 | 중앙 집중식 |
| **기술** | RFID + ANPR 선택적 | RFID 필수 |
| **미결제** | JPJ 위반금 연계 | 통행료채권 회수 |
| **API** | TOC, BOS 등 외부 연동 | 폐쇄형 |

### 2. PLUS (Malaysia)와의 비교

| 항목 | PLUS (기존) | Malaysia BOS (신규) |
|------|------------|-------------------|
| **차선 수** | SLFF 또는 유인 | SLFF/MLFF 혼용 |
| **실시간 처리** | 낮음 (유인 방식) | 높음 (10,000 TPS) |
| **데이터 통합** | 분산적 | 통합 (BOS 중심) |
| **AI 활용** | 제한적 | 광범위 (Text-to-SQL, RPA 등) |
| **거버넌스** | 단순 | 복합 (JVC, TOC, BOS, Clearing Center) |
| **확장성** | 낮음 | 높음 (멀티 JVC 지원) |

## 용어 사전

### A. 조직/역할

| 용어 | 약자 | 설명 |
|------|------|------|
| **Joint Venture Company** | JVC | 도로 운영권을 보유한 사업 주체 |
| **Traffic Operation Center** | TOC | 차량 통행 및 도로 운영 실시간 관리 |
| **Back-Office System** | BOS | 거래, 계정, 청구 관리 시스템 |
| **Clearing Center** | CC | 결제 기관 간 자금 정산 |
| **Toll Center** | TC | JVC 산하 지역 운영 센터 |
| **Toll Plaza** | TP | 실제 요금소 (Entry/Exit 지점) |

---

### B. 기술용어

| 용어 | 약자 | 설명 |
|------|------|------|
| **Radio Frequency Identification** | RFID | 800MHz 전파를 이용한 태그 인식 |
| **Automatic Number Plate Recognition** | ANPR | 카메라 기반 번호판 자동 인식 |
| **Optical Character Recognition** | OCR | 이미지의 문자를 자동 인식 |
| **Single Lane Free Flow** | SLFF | 단일 차선 무정지 통행 |
| **Multi-Lane Free Flow** | MLFF | 다중 차선 무정지 통행 |
| **Transaction Per Second** | TPS | 초당 거래 처리 건수 |
| **Application Programming Interface** | API | 외부 시스템 연동 인터페이스 |

---

### C. 거래/결제

| 용어 | 설명 |
|------|------|
| **Channel A** | RFID/ANPR 기반 BOS 직접 징수 |
| **Channel B** | TnG 연동 실시간 결제 |
| **Entry** | 고속도로 진입 지점 |
| **Exit** | 고속도로 퇴출 지점 |
| **Session** | Entry에서 Exit까지의 주행 기록 |
| **Unpaid Transaction** | 미결제 거래 (결제 미완료) |
| **Violation** | 위반 거래 (JPJ에 통지) |
